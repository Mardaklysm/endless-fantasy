# Gameplay Systems

## Overworld Movement

- The world is a 64x40 tile grid generated by `src/world/worldGenerator.ts`.
- New Game creates a fresh random world seed. Saves persist `worldSeed`, and Load Game regenerates the exact same world from that seed.
- Player position is `worldPos`.
- Movement is continuous pixel/tile-space movement with arrow keys/WASD.
- On directional input, the scene advances persistent visual coordinates every update and derives logical tile positions only when the player crosses tile centers.
- Holding a direction moves smoothly; releasing can stop between tiles rather than completing a forced tile step.
- Shift keeps faster exploration movement, but still advances continuously rather than skipping tiles.
- Visual position and logical tile position are intentionally separate during movement. Do not recompute rendering from stale logical tile coordinates mid-move.
- Terrain uses the 80 tile IDs in `src/data/worldTiles.ts`, grouped by biome, walkability, movement cost, and tags.
- World collision uses centralized `isWorldTileWalkable(tileId)` from the tile manifest.
- Water, river, ocean, swamp water, cliffs, lava, and blocked mountain tiles are not walkable.
- Bridges, roads, paths, and carved POI footprints are walkable.
- Boat and skyship flags still gate progression/location access, but they no longer make raw water or blocked mountains walkable.
- The generator creates coherent biome regions, rivers, bridges, roads, and procedural POI coordinates, then validates that required POIs are reachable from the start.
- Overworld locations use larger 3x3-ish footprints for rendering and entry. The terrain under each footprint is road/passable so the player can approach and enter naturally.
- Location footprints enter towns/dungeons/gate automatically when the player crosses into the footprint.
- Town south gates can exit when stepping into the exit tile or pressing south while already standing on the gate tile.
- Town interactions check the facing tile first, then the nearby forgiving service/NPC fallback.
- Menus, dialogues, and battles block exploration movement.

## Random Encounters

- Random encounters occur on dangerous world terrain and inside dungeons when settings allow.
- Encounter setting can be toggled in the menu.
- Encounter counter decrements only when the derived logical tile changes; forests/hills are more dangerous.
- World encounter tables are selected from generated tile encounter families such as plains, forest, hills, sand, water, and final/darkland. Roads and bridges suppress random encounters.
- Dungeon encounter tables are defined per dungeon.
- F9 debug menu can start an encounter for testing.

## Battle System

- Battles use individual initiative turns.
- Commands: Attack, Magic, Item, Defend, Run.
- At battle start and at the start of each initiative cycle, every living party member and enemy rolls initiative.
- Current formula: party initiative is `speed * 10 + random 0-12 + floor(luck / 2)`; enemy initiative is `speed * 10 + random 0-12`.
- Highest initiative acts first.
- Only the current party member gets a command menu.
- Player attacks, spells, items, defend, and run attempts pass through a short resolving animation on that actor's turn.
- Enemies execute AI on their own initiative turns and also use the short resolving animation path.
- After every living combatant in the current order has had a turn or been skipped, a new initiative cycle is rolled.
- The game does not use an all-party action queue round system.
- Battle presentation now uses v2 scene backdrops, enemies on the left, normalized fighter/priest/wizard class-sheet party battlers on the right, and lower panels for target/log, commands, and party status. The small redundant party head portraits are not drawn on the battlefield.
- Party attack animations use the class-sheet `attack_windup_left` then `attack_release_left` frames during the existing lunge timing. Battle idle uses the left-facing walk frame.
- Battle backgrounds are selected by dungeon first, then overworld terrain for random encounters.
- Enemy actions are randomly selected from each enemy's move list.
- Dead attack targets retarget to a living enemy.
- Boss battles cannot be fled.
- Defend is applied immediately and lasts until that character's next turn starts.
- Poison ticks at the start of the affected actor's turn.
- Sleep/stun are checked at the start of the affected actor's turn and skip that turn when active.
- Ward/starveil/silence durations tick at the end of the affected actor's own turn.
- Victory awards XP and gold; all party members receive XP by default.
- Game over occurs when all party members fall.

## Shops, Inns, And Clinics

Each town has item stock, weapon stock, armor stock, spell stock, inn price, clinic price, NPCs, and optional arrival behavior.

- Inns restore HP/spell charges and save.
- Clinics revive fallen characters for gold.
- Shops buy items, gear, and spells.

Shop UI is menu-based text only.

## Dungeons, Bosses, And Progression

Dungeons are data-driven in `dungeons()`:

- Moss Cave -> Root Relic.
- Ashen Keep -> Flame Relic.
- Tide Shrine -> Tide Relic.
- Skyglass Tower -> Gale Relic.
- Eclipse Spire -> final boss.

Each dungeon has two generated floors, random encounters, chests, a switch/gate puzzle, a boss tile, boss intro text, and reward text.

Dungeon floors, walls, gates, stairs, exits, chests, switches, and boss relic seals render from Batch 001 PNG tiles when available. Town floors/walls/service props now render from `assets_v2`. Opened chest art is wired through `openedChests`.

Progression flags:

- Root opens Brinewick boat route.
- Boat enables Tide Shrine route.
- Flame and Tide help unlock Skyglass Tower.
- Four relics open Starfall Gate.
- Gate/skyship allow Eclipse Spire.

## Save/Load

Saves use browser `localStorage`.

Saved data includes:

- Party state
- Inventory and gear bag
- Gold
- Positions
- Current town/dungeon/floor
- Story flags
- Opened chests
- Puzzle flags
- Defeated bosses
- Settings
- Encounter counter

Known gap: save data has no explicit version field or migration strategy yet.

## Known Gaps

- No automated tests.
- No visual spell/attack animation timing system yet.
- Mouse menu support is minimal.
- Item/equipment/relic icons are loaded but not yet displayed in menus/HUD.
- UI panel skin now uses the v2 compact window frame with live Phaser text; reference UI windows with sample text remain source-only.
- Save schema migration/versioning is not implemented.
- Balance has only manual smoke-test coverage so far.
